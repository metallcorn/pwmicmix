## Дополнительный функционал: мониторинг и управление в реальном времени

### Внешний вид
Тёмная тема (фон #111), вертикальные каналы как на физическом пульте.
Мокап интерфейса прилагается — реализовать максимально близко к нему.

### Backend: аудио-анализ

GET /api/levels
  - Опрашивается фронтом каждые 80мс (WebSocket или polling)
  - Для каждого активного виртуального микрофона возвращает уровень сигнала (0.0–1.0)
  - Реализация: запускать pw-record --target={node_name} --channels=1 --rate=44100
    в subprocess, читать PCM-байты, считать RMS за последние 50мс
  - Формат ответа: { "mic_1": 0.62, "mic_2": 0.91, "mic_3": 0.18, "mic_4": 0.0 }
  - Если устройство не отвечает / сигнала нет 2+ секунды → статус "no_signal"

POST /api/volume
  - Body: { "mic_id": "wing_virtual_mic_1", "gain": 0.85 }
  - Применяет программный gain через: pw-cli set-param {node_id} Props '{"params": ["Props", {"volume": 0.85}]}'
  - Gain хранить в памяти, применять при восстановлении сессии

### Frontend: компоненты пульта

1. Канальная полоса (одна на каждый активный маршрут):
   - Название (пользовательское, например "Ведущий")
   - Источник (например "WING ch3" или "Logitech FL")
   - VU-метр вертикальный: зелёный 0–70%, жёлтый 70–85%, красный 85%+
     Мигающий красный индикатор "clip" при превышении 92%
   - Вертикальный фейдер (input type=range, writing-mode: vertical-lr)
     значение отображается в dB (−∞ при 0, 0dB при максимуме)
   - Маршрут: "WING ch3 → Mic 1" маленьким текстом
   - Тумблер вкл/выкл канала (toggle)
   - Статусная пилюля: "сигнал" (зелёная), "клиппинг!" (жёлтая мигающая),
     "нет сигнала" (красная), "выкл." (серая)

2. Мастер-канал (левый, всегда виден):
   - Общий VU показывающий суммарную активность
   - Мастер-фейдер (применяет gain ко всем каналам)

3. Топ-бар:
   - Индикатор статуса (зелёная точка с анимацией пульса = активно)
   - Количество активных микрофонов
   - Кнопка "⏹ Стоп" (красная, большая)

4. Нижняя строка:
   - Имя устройства-источника, количество каналов
   - Кнопка "+ добавить канал" (открывает wizard выбора)

### Диагностика и алерты
  - Если канал включён но RMS = 0 дольше 3 секунд → статус "нет сигнала",
    канал подсвечивается красной рамкой
  - Если канал клипирует (>92%) дольше 500мс → жёлтый алерт "клиппинг!"
    с рекомендацией "снизить уровень на пульте"
  - Toast-уведомления (2 сек, снизу справа) при: подключении устройства,
    ошибке линковки, успешном создании маршрута

### Веб-интерфейс (wizard для добавления каналов)
  Если нет активных маршрутов ИЛИ нажата "+ добавить канал":
  - Модальное окно поверх пульта
  - Шаг 1: список устройств (человекочитаемые имена)
  - Шаг 2: чекбоксы каналов + поля для имён
  - Шаг 3: подтверждение → POST /api/start → закрыть модал, показать новые каналы

### Технические детали реализации уровней
  Использовать parec (PulseAudio record, работает с PipeWire):

  parec --device={node_name}.monitor --channels=1 --rate=8000 --format=s16le

  Читать поток в отдельном потоке (threading.Thread) для каждого микрофона,
  накапливать буфер 80мс, считать RMS:
  import numpy as np
  rms = np.sqrt(np.mean(np.frombuffer(buf, dtype=np.int16).astype(np.float32)**2)) / 32768


  ## Дополнение: управление каналами на лету

### Каждый канал имеет:
- Кнопка ⚙ — открывает модал переназначения источника/порта.
  POST /api/reassign { mic_id, new_device, new_port }
  Backend: убивает старый pw-loopback, убирает pw-link, создаёт новый
- Кнопка ✕ — немедленно удаляет канал.
  POST /api/remove { mic_id }
- Двойной клик по имени канала — инлайн input для переименования
  (только UI-имя, не пересоздаёт PipeWire-ноду)

### Кнопка "+ канал" (в топбаре и в конце стрипа)
  Тот же 2-шаговый wizard: устройство → порты+имена → POST /api/start
  Новые каналы добавляются в конец стрипа без перезапуска остальных

### Peak hold на VU-метрах
  JS: пиковое значение затухает со скоростью 0.98 за кадр (≈ 2 сек до нуля)
  Отрисовывается отдельной div-полоской поверх VU-бара
